Marking Scheme and Appeal Procedure

Grade Checking

  • We have uploaded all test cases to ZINC and regraded your latest submissions.
  • Individual test case results and scores can be found in the latest PA3 grading report on ZINC. The total number of given and hidden test cases is 100, resulting in a base score of 100 * (1 + 0.25) = 125.
  • Please also check your "total score" HERE, which includes the ZINC normalized score along with any applicable penalties (e.g., late submission penalty). The formula is as follows: 
    "PA3 total score" = "ZINC Score" / 1.25 - "Penalty (number of minutes late after 00:59)"
  • Please double check that your ZINC score shown on the above lookup page is consistent with the ZINC score you received.
  • We will perform plagiarism checking at a later time and marks will be adjusted for those who are caught.
  • Statistics:
    • Mean: 89.90
    • SD: 20.32
    • Max: 100.00
    • Median: 99.00


  • If you want to appeal your grade: you MUST read this whole page and follow all procedures correctly. Your appeal will be dropped otherwise.

Marking Scheme

  • There are in total 100 test cases, each is run once with an executable compiled normally and once with an executable compiled with fsanitize for checking memory leak and address issues.
  • The test cases are numbered as #XY. Test case #1XY is the fsanitize version of test case #XY (except for #100 and its leak-checking version #200).
  • Each of the test cases run without memory leak checking is worth 1 mark. The second run of each test case with memory leak checking is worth 0.25 mark. Therefore, the base score is calculated as 100 * (1 + 0.25) = 125.
  • The given test cases are also included in the final testing (#1 to #50).
  • There is no partial credit.
  • Your program is given at most 30 seconds to run for each test case.

Appeal Guidelines and Procedure

1. Appeal Guidelines

  • Runtime errors, crashes, and inconsistent output due to incorrect memory access / memory allocation / memory deallocation / missing variable initialization / missing return values / out-of-bound array access / etc. may not occur all the time on all machines. However, even you may not encounter the error/crash when you test your program, if your program crashes during any of our testing on ZINC, the corresponding test cases will be considered as failed and will receive 0 marks as your code is indeed faulty.
  • If you somehow get a correct output on your own computer but you get a different output on ZINC, it is likely your code is faulty. Refer to the previous comments for possible causes and Piazza posts for similar issues. Verify your code to make sure they aren't faulty and won't produce any runtime error before you appeal those cases. Compare your solution with the sample solution yourself if needed.
  • We can run the test cases again for you (after you have verified them yourself), but all grading shall be done objectively in an automatic and consistent way on the very same machine (the ZINC grading machine, not any other machine including yours or ours). However, if your program crashes or gives unreliable results, it is guaranteed that the problem is somewhere in your code, and marks will not be given. For grading, only the result of your program on the ZINC machine matters.
  • It is possible that you get a lower mark as we rerun the test cases for you and found additional crashes/mistakes. We rerun all test cases (not just the ones you want to rerun) for all appeals. So proceed carefully at your own risk, not just blindly appeal your grade anyway.
  • It is mentioned in the assignment instruction that "It is required that your submissions can be compiled and run successfully in our online autograder ZINC. If we cannot even compile your work, it won't be graded. Therefore, for parts that you cannot finish, just put in dummy implementation so that your whole program can be compiled for ZINC to grade the other parts that you have done."
  • For compilation errors, you get 0 mark. To appeal, you should find out how to fix the compilation errors yourself, and then clearly tell us how in your appeal statement. If the fix is simple, we will help you fix it according to your instructions then regrade it with a penalty. The fix should fix compilation errors only and must not change the logic of your program.
  • Please understand that, for fairness, we usually do NOT allow any student to change their submitted code. On very rare circumstances, say, adding a character will make you get back at least 50 marks, then we may consider it. Even then, a major penalty will apply to the regraded result.
  • If you have any questions about a test case, please post on Piazza so that other students can see. Remember to use the "pa3" tag.

2. Appeal Procedure

  • If you are certain that there is a grading mistake after thoroughly reviewing this page, the assignment description/FAQ, and running the test cases yourself, submit a PDF or Word file to the "PA3 Appeal" submission box under Canvas Assignment. Ensure that the submitted file contains the following information in order for your appeal to be processed. Please note that appeals with incomplete information will not be processed.
    • Your HKUST ITSC username
    • Your student ID number
    • Test cases for which you want to appeal
    • Proposed code changes, and/or other justifications. Please note that there will typically be penalty points for code changes after appeal.
    • Do not attach any .exe/executable file.
  • Appeal deadline is May 21st (Tuesday) 23:59. No late appeals will be processed.

3. After Appeal

  • All appeal results will be released after the submission deadline. The grading feedback on Canvas will include comments on the results of the appeal, specifying which code changes have been accepted or which test cases have been successfully appealed.
  • If your appeal case is accepted, we will modify your code accordingly and rerun it on ZINC. However, if your appeal case is not accepted, no further action will be taken.
  • The finalized total score will be updated based on the number of test cases correctly passed in the latest program run on ZINC. If you choose not to submit an appeal or if your appeal requests are not approved, your score will remain unchanged. However, if your appeal is approved and code changes are made, resulting in a regrading on ZINC, your score may change. Typically, an approved appeal will result in an increased score, but there is also a possibility of a decreased score, as outlined above.
  • Once all appeal cases have been reviewed, a Canvas announcement will be made, and you can check your scores HERE again for any updates.
