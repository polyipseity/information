Lab 4 Order Management

Overview

In this lab, we practice inheritance by managing store orders. We think an order is essentially a product collection, which can be derived into an ordinary customer order or a membership order.

Based on this, we first need to complete a base class ProductCollection and then derive the OrdinaryOrder and MemberOrder classes.

To focus more on inheritance, we provide basic implementation of the classes. For functions not fully described below, please refer to Lab Tasks and skeleton code for better understanding.

First, ProductCollection is the base class. insert and remove insert and remove a product to the product array, respectively. They return 0 if the operations are executed successfully, 1 otherwise. contain checks if its input product exists or not in the product array and returns true or false, respectively. 

/* ProductCollection.h */
class ProductCollection {
private:
  string collectionID; // ID of the collection
  int* products; // dynamic array of product IDs
  int capacity; // capacity of the product array
  int size; // current number of products (integers) in the product array
public:
  ProductCollection(const string& collectionID, int capacity);
  ~ProductCollection();
  string getCollectionID() const; // returns collectionID
  int getCapacity() const; // returns the product array capacity
  int getSize() const; // returns current number of products in the product array
  int readProduct(int i) const; // returns i-th product in the product array
  void printProducts() const; // prints out all products in the product array
  int insert(int productID);
  bool contain(int productID) const;
  int remove(int productID);
  void increaseCapacity(int additionalSpace); // TODO 1
};

OrdinaryOrder represents orders made by ordinary customers. It is directly derived from ProductCollection. 

/* OrdinaryOrder.h */
enum OrderType {ON_SITE, PICK_UP, COURIER}; // enumeration of delivery type
class OrdinaryOrder : public ProductCollection {
private:
  string name; // customer name of this order
  OrderType type; // delivery type of this order
public:
  OrdinaryOrder(const string& orderID, const string& name, OrderType type, int capacity); // TODO 2.1
  int addFromOrder(const OrdinaryOrder* anotherOrder); // TODO 2.2
};

MemberOrder represents orders made by membership customers. It is directly derived from OrdinaryOrder, indirectly derived from ProductCollection. 

/* MemberOrder.h */
class MemberOrder : public OrdinaryOrder {
private:
  float rewardPoints; // reward points the products in this order can gain, 0.5 point for each product
public:
  MemberOrder(const string& orderID, const string& name, OrderType type, int capacity); // TODO 3.1
  float getPoints() const; // returns reward points of this order
  void insert(int productID); // TODO 3.2 (we do not use dynamic-binding and polymorphism here)
  void remove(int productID); // TODO 3.2 (we do not use dynamic-binding and polymorphism here)
  void addFromOrder(const OrdinaryOrder* anotherOrder); // TODO 3.3
};

Lab Tasks

Task 1: ProductCollection class
  • TODO 1:
    increaseCapacity in ProductCollection.cpp increases the capacity of products array. To complete it, you need to allocate a new integer array, and the capacity of the new integer array should be the sum of capacity and additionalSpace.
    We assume the value of additionalSpace is greater than 0 here. Please pay attention to memory leakage. The products in the product array should remain the same before and after the function.
Task 2: OrdinaryOrder class
  • TODO 2.1:
    To complete the constructor of OrdinaryOrder in OrdinaryOrder.cpp, you need to use ProductCollection constructor and initialize name and type.
  • TODO 2.2:
    addFromOrder in OrdinaryOrder.cpp allows adding products to current order from another ordinary customer order. To compelete it, you first need to check if the size of anotherOrder will exceed the remaining capacity of current order. If it will, do not insert any product and return 1. If the capacity is enough, add products from anotherOrder to the product array of current order, and return 0.
Task 3: MemberOrder class
  • TODO 3.1:
    To complete the constructor of MemberOrder in MemberOrder.cpp, you need to use OrdinaryOrder constructor and initialize rewardPoints with 0.
  • TODO 3.2:
    For the derived MemberOrder class, we need to update the reward points every time we insert or remove a product. Note that dynamic-binding and polymorphism are not used here.
    To complete insert in MemberOrder.cpp, you need to call insert from ProductCollection class first. If the insertion is successful, we add 0.5 to rewardPoints. To complete remove in MemberOrder.cpp, similarly, we need remove from ProductCollection and decrease rewardPoints by 0.5 if the remove is successful.
  • TODO 3.3:
    addFromOrder in MemberOrder.cpp allows adding products to current order from another ordinary customer order (no need to consider adding from another member order here). To compelete it, you can call addFromOrder from OrdinaryOrder, and if the insertion is successful, update rewardPoints. Same to TODO 3.2, each product in products equals to 0.5 point.

Compile and Test

We have prepared a makefile in the provided skeleton code. After you have finished all tasks (Task 1 ~ 3), you can test your program by typing make in the terminal to compile the files. Then you can run the program by typing ./lab4.exe in the terminal. Your program must be able to re-generate exactly the same output as the sample output that has been shown in the overview section. You should read the main.cpp file carefully to understand how the sample output is generated.

Deadline and Guidelines

The lab assignment is due 10 minutes after the end of your lab session.
We will use an online grading system, ZINC, to grade your lab work. Please upload the following files as a .zip file to ZINC: 
- ProductCollection.cpp
- OrdinaryOrder.cpp
- MemberOrder.cpp
Note that you do not need to submit the makefile.
Please remember to zip the files directly, NOT zip a folder with the required files. After submitting, please review the ZINC report to make sure your submission is parsed correctly. You can make multiple submissions to ZINC before the deadline. Only the last submission will be graded.

Note: The ZINC score report contains 5 test cases numbered #1 ~ #5. There are 5 additional test cases #11 ~ #15, which are the same as the first 5 test cases, but the program is compiled using the -fsanitize flag. You will lose points for these test cases if your program contains memory leak or other runtime issues.
