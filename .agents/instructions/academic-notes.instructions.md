# Academic notes instruction

This instruction file surfaces the essential, quick-reference guidance from the `academic-notes` skill for agents and maintainers working on course folders under `special/academia` (including the private mirror path `private/special/academia`). **Agents must read every file in `.agents/skills/academic-notes/` and every line of those files before acting; do not assume the concise guidance is exhaustive.** It is intentionally concise — consult the full skill docs for examples and tooling.

## Scope & purpose

- Applies to: `special/academia/**` and mirrored `private/special/academia/**` course folders.
- Purpose: give agents a compact checklist for authoring, validating, and triaging course notes using the `academic-notes` conventions and helpers.

## Quick guidance for agents

- **Adding new lecture content**: Before creating new topic notes, check whether the new content (1) **duplicates** existing content → add section links only; (2) **enhances** existing content → enhance that note (prose + flashcards) and add section links; (3) has **no or little overlap** → then add new note(s). Prefer one place per concept; avoid duplicate or overlapping notes. If a lecture introduces a coherent cluster of related ideas that deserves a durable home, proactively suggest or create a topic-specific note and route the session to it with § links.
- **Large question collections**: When a single questions page grows large (for example several tutorial weeks plus review prompts in one file), convert it to folder format: create `questions/index.md`, split the material into child pages such as one file per tutorial week and one file per review/exam block, update all incoming links, and remove the obsolete monolithic page so there is only one canonical questions location.
- **Question-family splits**: If official question sources come in distinct families (for example tutorials, problem sets, exam practice, or solution sheets), keep each family as its own set of child pages inside `questions/` rather than squeezing everything into the tutorial pages. Update `questions/index.md` so the folder remains the single canonical navigation point for all official and repository-authored question material.
- **Appendix or auxiliary PDF handouts**: If a course appendix/handout contains substantive theory or worked examples that the course notes rely on, do not leave the content stranded behind a removable PDF link. Absorb the material into topic-specific notes inside the course folder, then let `index.md` point to those notes. PDF links may remain only as optional attachments; they should not be the sole durable home or navigation target for required content.
- **Theorem hypotheses from appendices/handouts**: Do not blindly preserve theorem statements from slides or appendices if a hypothesis is missing or overstated. When importing classical analysis/probability results into course notes, verify the exact assumptions needed in the current setting (for example, Riemann-integrability of a derivative in FTC II) and repair the statement in the note rather than copying a false or overbroad version.
- **Theorem conclusions and strength**: When restating a theorem, prefer the standard or minimal conclusion in the headline statement, and mention stronger consequences separately when they follow in the present setting. For example, in FTC I it is standard to say the accumulated-area function is uniformly continuous; one may then add that it is actually Lipschitz here because $|F(x)-F(y)|\le \sup|f|\,|x-y|$. Avoid replacing a standard theorem statement by a strictly stronger but less standard version unless the stronger form is genuinely the course's intended theorem.
- **Analysis-style appendix notes**: When absorbing calculus or analysis appendix material (for example Riemann or multiple integrals) into course notes, do not stop at formal definitions and formulas. Add geometric or probabilistic interpretation as well: box approximations, slicing intuition, indicator-function “on/off” viewpoints, and event-region interpretations for probability calculations. This is especially important in honors probability notes, where conceptual interpretation should accompany the formal statement.
- **Basic definitions in higher-dimensional analysis notes**: When introducing multidimensional objects such as rectangular sets, diameter, volume, lower sums, or upper sums, add both the geometric meaning of the definition and an explicit comparison with the one-dimensional case. Readers should be able to see not only the formula, but also which 1D idea is being generalized.
- **Worked examples in analysis-style notes**: When including explicit integral calculations, do not jump straight from the setup to the final answer. Show the main steps briefly in both the prose and the flashcards, so the reader can see the calculation strategy (which variable is integrated first, what is treated as constant, and what simplification occurs at each stage).
- **Combining short concept cards and short worked steps**: If a concept’s definition and intuition are both short and naturally belong together, prefer a single combined flashcard over two mechanically separated ones. Likewise, if an example requires only a few brief computational steps, prefer one answer card that contains both the short derivation and the final value rather than splitting “steps” and “answer” into separate cards.
- **Lab-preparation slides and lab manuals (e.g. “Prepare for lab N”, `LabN_2026s.pdf`)**: Treat lab-prep/tutorial decks and lab handouts as normal lecture ingestion. Centralise definitions, worked examples, and circuit behaviour (e.g. resistor ladders, diode/LED safety, Zener regulation, transistor switches, H-bridges, motor drivers, LM7805 usage) in topic notes, then point to those sections from the relevant lab or tutorial via § links and a small number of workflow/safety flashcards. Avoid duplicating entire worked examples or long procedures in weekly index entries; the index should primarily route readers to topic notes.
- **Validation suppressions**: Prefer fixing content over suppressing rules. When suppression is genuinely needed, use one of three forms with a clear rationale: `ignore-line[rule_id]`, `ignore-next-line[rule_id]`, or `ignore-file[rule1, rule2]`. Line and next-line suppressions apply only to their specific line; file-level suppressions apply to all messages for the listed rules in that file and are considered redundant (and rejected) if the rule never fires anywhere in the file. File-level suppressions are primarily intended for special cases like assignment-style lab index pages that intentionally omit the standard `# index`/`children` shell while keeping the same rules active for the course’s main index.
- Do NOT create or edit `general/` files automatically. Use the Wikipedia helper only to discover canonical titles; if it implies a `general/` path, ignore that for course-note placement. Topic-specific course notes still belong under `special/academia/<INSTITUTION>/<COURSE>/` with lowercase filenames unless a proper noun requires otherwise.
- Require a flashcard activation tag in course files: `flashcard/active/special/academia/<INSTITUTION>/<PAGE>` (case-insensitive). The validator will flag missing tags. Notes may use any of the three markup types – cloze `{@{}@}`, two-sided `::@::`, or one-sided `:@:` – but remember that the latter two must stay on a single line.  **Do not trim prompts for calculations**; the warning rules exist to catch missing numeric/symbolic data, not to enforce brevity.  You may write arbitrarily long equations or value lists before the separator; splitting a single logical example into multiple cards just to shorten the left-hand text is counter‑productive. LaTeX on the **left-hand side** of a flashcard is perfectly acceptable when it is the natural way to state the prompt; do not rewrite a mathematically natural prompt into weaker prose merely to avoid left-side LaTeX. For **derivation or approximation clusters**, though, prefer a descriptive left-hand prompt naming the step or role (for example `Poisson approximation / exact factorization`, `hypergeometric approximation / intuition`) and keep the formulas on the right-hand side. When converting prose or examples into flashcard bullets, also copy any relevant diagrams or images from the paragraph above and include them in the question text – the validator will remind you if an image appears in the source but not in the card. A suppression directive is permitted only when the card truly tests a conceptual fact (e.g. `<!-- check: ignore-line[two_sided_calc_warning]: conceptual -->`).
- Make every flashcard **standalone and understandable in isolation**. Do not rely on neighbouring cards to supply hypotheses, notation, or conditions: restate local assumptions such as $f \ge 0$, $P[B] > 0$, the sample space, or whether sampling is with or without replacement when they are needed to make the card precise.
- Prefer **single-focus flashcards** for conceptual material: if a card answers multiple distinct points, split it into multiple smaller cards. Keep calculation cards self-contained on the left-hand side (do not delete givens just to shorten prompts). If nested proof or intuition steps help, encode them inline with `<br/>` and labels like `(1)`, `(1.1)`, `(1.2)`, `(2)` rather than using a markdown sublist.
- If you use **nested flashcards** in any academic-notes file, indent them by exactly two spaces per level and make each nested QA prompt repeat the full parent path rather than relying on indentation for context. For derivation or approximation clusters, keep the left-hand prompt descriptive, but it is still good to show the key mathematical terms or limit expressions on the right-hand side so the strategy is explicit rather than purely verbal.
- When embedding LaTeX, always use `$...$` for inline math and `$$...$$` for display equations; do **not** employ `\(\)` or `\[\]` variants, as the validator and rendering pipelines expect dollar‑sign delimiters. **Accounting courses:** do **not** use LaTeX; use plain text, numbers, and symbols (×, −, ÷).
- When adding comments, avoid placing more than one directive of the **same type** on a single line; the validator now flags duplicates and authors should merge them into a single comment listing all applicable rule IDs (the syntax already supports commas).
- Use the validator conservatively: run `check.py --content` for advisory guidance; it will flag missing tags, exams before sessions, duplicate week numbers, unscheduled sessions carrying topics, out-of-order semester headings, and similar structural issues. Treat its output as suggestions unless maintainers request strict enforcement.
- Treat submodules (including `private/`, `tools/pytextgen/`, `tools/pyarchivist/`) as read-only unless the user explicitly grants permission.
- Prefer small, reviewable changes to skill docs and helper scripts; document rationale and link to the Continuous improvement section of SKILL.md when proposing edits.
- Unless the user explicitly asks about commits or version-control steps, do not add commit reminders or commit-oriented instructions during academic-notes work; keep the guidance centered on content, structure, validation, and documentation quality.
- **Always run the `check.py` validator after using any edit tool.** The agent frequently emits malformed Markdown, so validating immediately helps catch and fix errors or warnings before they land in a PR.
- Treat topic-note maintenance as a three-part update: if you add, remove, split, merge, or rename content in a topic note, update the corresponding flashcards in that note and then update every affected `index.md` section link or anchor in the same task.

## Course index and logistics (conventions)

- **Staff and contact**: Do not put instructor or TA names or email addresses in course notes; refer to the official syllabus or LMS for contact. Office hours (day, time, format) may be included in logistics.
- **Separator**: After the course list (institution, `name:`, `credits:`), insert `---` before the course description.
- **Sections format**: Under `logistics` → `sections`, use one key for the chosen stream (e.g. `lecture: L1`). List all section identifiers and their details under that key (e.g. L1, L2, L3 with venue and times). Do not create separate `lecture: L2`, `lecture: L3` keys.
- **Chosen tutorial/lab stream drives metadata**: If the chosen stream under `sections:` is, for example, `tutorials: T1B`, then weekly tutorial entries should use T1B’s real venue and time in `datetime:`/`venue:`. Do not accidentally copy another stream’s slot metadata into the weekly timeline. If the selected slot is canceled for scheduling reasons caused by another stream, prefer `status: no class` unless the selected slot itself falls on the public holiday.
- **Session headings**: Use `## week N lecture` and `## week N lecture 2` (space in “lecture 2”). Status has no bearing on the heading—use the same type as the slot (e.g. week 3 lecture); do not use week 3 no class or put holiday in the heading (e.g. `## week 3` not `## week 3 (Lunar New Year)`).
- **No-class days**: Omit `topic:` for sessions without classes. Use `status: public holiday: <name>` when the holiday is known (e.g. Lunar New Year, Labor Day); use `status: no class` for other non-teaching days (e.g. midterm break).
- **Index session flashcards**: Omit session-level flashcard bullets in the index when they only describe scope or coverage; keep substantive flashcards in topic notes and questions. When a session introduces a reusable recognition pattern (e.g. an “infinite” resistor ladder method, or reading Zener clamp breakdown from a $V_Z$–$V_{\text{in}}$ plot), document the general method once in the relevant topic note (with self-contained QA cards) and use brief session-level cards in the index to point students to that method rather than re-deriving it.
- **Emphasis**: Use underscore for italic and bold: `_italic_`, `__bold__`. Do not put LaTeX inside emphasis; keep math outside `_..._` and `__...__`.

## Short author checklist

1. Add YAML frontmatter including `title`, `aliases`, `tags:` (must include the `special/academia/<INSTITUTION>` tag).
2. Add or confirm flashcard activation tag in `tags:` when flashcards are desired.
3. Ensure `index.md` pages contain `# index` and a `children:` or `## children` section where appropriate. For auxiliary index pages such as `labs/index.md` and `labs/lab N/index.md`, you may either keep a minimal index shell (e.g. `# index` plus a minimal children list) or, when the page is really an assignment note (e.g. lab summary-sheet submissions), drop the visible shell and use a file-level suppression such as `<!-- check: ignore-file[index_heading_rule,index_children_rule]: assignment-style index -->` so the validator does not enforce the standard index heading/children rules on that file. Do not add suppressions for rules that never fire in the file; the validator flags such redundant directives as errors. For courses with labs, each timed lab session in the main course index (e.g. “week 4 lab 1”) should both (a) list § links into the relevant topic notes and (b) include a bullet linking to the corresponding assignment-style note under `labs/lab N/index.md` so readers can jump directly from the timeline to either the conceptual material or the Canvas-style lab page.
4. For weekly pages, include `datetime:` ranges and concise `topic:` / `learning_outcomes:` or `takeaway:` entries. In **children**, list each topic note as a single link; do not nest § links there. In **lecture, tutorial, and lab session entries**, list each topic note covered as a parent link with `- topic name / [§ section name](file#anchor)` per section underneath. For **index-only content** (e.g. session-level flashcards not in topic notes), keep the course code as parent and nest those bullets under it. For glosses that contain bullet-like numbered items, keep them on a single line using ` <br/> ` between items (for example, `(1) … <br/> (2) … <br/> (3) …`), rather than turning them into a markdown list. Use percent-encoded spaces in anchors, or hyphenated anchors for headings with commas/special chars. When you merge or remove ### subsections in a topic note, update the § links in the session entries to match. More generally, whenever you add, remove, split, merge, or rename topic-note sections, sync the affected `index.md` links before you consider the edit complete.
5. **Topic notes**: For course topic-specific notes **other than** `journal entries.md`, use only **two-sided** (::@::) or very rarely **one-sided** (:@:) flashcards; **do not use cloze** ({@{}@}) in the prose—add more QA cards instead. Cloze is only for journal-entry examples. Use lowercase filenames (e.g. `voltage regulator.md`), or the full canonical term when it matches the general article (e.g. `brushed DC electric motor.md`). In general, prefer the **singular** canonical concept name for a topic-specific note’s filename and H1 title (e.g. `cumulative distribution function.md`, `continuous distribution.md`, `discrete distribution.md`, `probability measure.md`), unless the concept is inherently plural by standard usage (e.g. `elementary combinatorics.md`). Aliases = general term and synonyms only (not instances like LM7805); **include both singular and plural forms** (e.g. brushed motor / brushed motors). When linking to `general/`, use **lowercase first word** in both link display text and path (e.g. `[brushed DC electric motor](.../general/brushed%20DC%20electric%20motor.md)`). Circuit examples must state topology, polarities, and loop direction; when introducing a component via an instance, define the general concept (e.g. IC, datasheet) with flashcards. Use ### subsections under ## where it improves hierarchy, and give each subsection its own `---` and “Flashcards for this section are as follows:” block. Avoid standalone “Examples” sections; instead, fold short worked examples into the relevant conceptual section’s prose and cards (e.g. the two-roll die example under the uniform distribution section). **Do not blindly copy topic-note section names or groupings from the source course materials** (slides, PDFs, LMS pages); design headings that express the underlying concept and its role (e.g. “conditional probability: definition and multiplication rule”, “independence via cards and coins”, “law of total probability and Bayes’ theorem”). When several named objects appear together in the source, split them into separate sections if each object gets its own real explanation (for example, keep “discrete uniform distribution” and “Bernoulli distribution” as separate headings rather than combining them). If a lecture note starts accumulating substantial material for a secondary concept (for example cumulative distribution functions inside a broader continuous-distribution note), prefer splitting that material into its own singularly named topic note and updating all affected index links rather than leaving a large mixed-concept page. Conversely, if a heading contains a canonical named law plus an explanatory model label, prefer the canonical law name as the section title and put the model interpretation in the prose instead (for example, “binomial distribution” rather than “binomial distribution and Bernoulli-trial model”). If a source heading combines a broad construction with one especially important canonical instance, prefer splitting it into a parent section plus a dedicated subsection for that instance when the hierarchy is clearer (for example, “generated sigma-algebras” with a subsection “Borel sigma-algebra”). For structural results like the law of total probability and Bayes’ theorem, explicitly state the hypotheses (e.g. what it means for $(B_j)$ to be a partition of $\Omega$) and include brief derivation sketches both in the prose and in a small cluster of focused cards (partition definition, law statement with derivation idea, Bayes formula, and countable extension). Describe **course-specific circuit layouts** explicitly (e.g. H-bridge: top row both PNP, bottom row both NPN) rather than a generic topology. For **accounting courses**, include a topic note `journal entries.md`: one journal entry type per section; each example in one blockquote (scenario, markdown table Dr/Cr right-aligned `|--|--:|--:|`, optional explanation/calculation); link all sections from the index and relevant week with § links. **Table header:** first column header = very brief description of the entry; wrap that description and **each account name** in the first column in clozes. In each example: mask **all** Dr/Cr amounts with clozes; add scenario (and calculation/explanation) clozes covering almost the full text but **leave a few hint words** at the start or end; for calculations, **cover both sides of an equality separately** and **cover the text (non-equation) portion** with hint words uncovered. **Cloze delimiter:** closing `}@}` before punctuation (e.g. `{@{text}@}.` not `{@{text.}@}`). Use **`&nbsp;`** for the thousands separator in amounts (e.g. 50&nbsp;000), not a comma. **Warranty:** settlement credit account = Cash, Inventory, or Accrued Payroll (not just Cash). **Service-type warranty:** revenue = performance obligation satisfied over the service period; in examples, service period typically starts after assurance-type warranty has expired.
   - In a course folder’s **topic-specific notes**, the course context, the filename, **and the note title** are already visible, so omit redundant filename/title text from flashcard prompts. Write local labels such as `definition`, `probability space`, or `Bayes theorem finite`, not `MATH 2431 / probability measure / ...`, not `probability measure / ...`, and not just `probability measure` as a top-level prompt. For nested flashcards, keep the full **in-file** parent path (for example `Poisson approximation / statement`). Use an explicit course-code prefix (`COURSE / topic / item`) only for **index/session-level cards** or other mixed-context pages.
   - For **honors courses** (for example when aliases or the course name include “Honors”, such as “Honors Probability”), think carefully and thoroughly about the underlying theory before adding, editing, or removing topic-note content; include clear arguments, derivations, or proof sketches for important results (not just bare statements), split long chains of reasoning into a few logically separate paragraphs instead of one over-dense block, and feel free to add more high-quality, conceptually rich flashcards than you would for a non-honors version of the course. For measure-theoretic constructions (e.g. generated $\sigma$-algebras), make existence/minimality details explicit, include short intuition for omitted deep proofs (e.g. interval-length nonextendability on $[0,1)$), and retain the course’s concrete examples in both prose and flashcards. For cumulative distribution functions, state and briefly prove the standard properties (range, monotonicity, right-continuity, limits at infinity, jump formula) rather than listing them only as facts. Also make explicit what it means for two distributions to be equal (equality of measures, equivalently equality of cumulative distribution functions on $\mathbb{R}$), warn against tempting but false shortcuts such as matching only atoms, only support, or only a few summary statistics, and make singular-continuous examples such as the Cantor distribution more concrete by giving a stage-by-stage construction and the associated intuitions for “no atoms” and “still singular.” When discussing densities, emphasize that equal cumulative distribution functions imply equality of densities only **almost everywhere**, not everywhere, and include at least one remark or flashcard explaining that densities may differ on Lebesgue-null sets while defining the same law. When the note reaches absolutely continuous cumulative distribution functions, do not stop at a vague slogan or at only one characterization: if the course material uses the equivalence, list the four probability/measure-theoretic formulations explicitly in prose and flashcards—(1) existence of a nonnegative measurable density $f$ with $P[X\in A]=\int_A f(x)\,dx$ for every Borel set $A$, (2) absolute continuity of the cumulative distribution function, (3) vanishing probability on every Lebesgue-null Borel set, and (4) the epsilon-delta small-Borel-set criterion $\lambda(A)<\delta\Rightarrow P[X\in A]<\varepsilon$. When stating item (2), also spell out what “$F$ is absolutely continuous” means, for example via the standard disjoint-interval epsilon-delta definition, rather than leaving it as a bare label. For misconception-clarification prose in honors probability notes, do not stop at a warning sentence: attach a very simple counterexample in both the prose and the flashcards (for example fair-coin variables with the same Bernoulli law but different pointwise values, uniform vs density $2x$ on $(0,1)$ for same-support / same-singleton-mass failures, or a trivial-sigma-algebra nonmeasurable function). For random-variable notes, use value-space notation that matches probability conventions: write $(E,\mathcal{E})$ or $(\mathbb{R},\mathcal{B}(\mathbb{R})))$ for the target measurable space, and explain explicitly that a symbol like $B\in\mathcal{E}$ is a measurable subset of the value space whose preimage $X^{-1}(B)$ must lie in $\mathcal{F}$. When introducing the countable-image viewpoint, explain the motivation rather than only the definition: if $\Omega_X$ is countable, then the law is determined by the point masses $P[X=x]$ on the actually attained values, so one can compress the original experiment to the smaller image space without losing anything relevant about $X$. When presenting the Lebesgue-Stieltjes construction, explain how the pseudoinverse idea is discovered from the inverse-transform heuristic, explicitly say what the random variable $X$ represents in heuristic expressions such as $U=F(X)$, explain why the ordinary inverse fails when the cumulative distribution function has jumps or plateaus, include both standard generalized-inverse formulas and a brief explanation of why they agree, and make that equivalence genuinely rigorous: isolate sets such as $A_u=\{y:F(y)<u\}$ and $B_u=\{x:F(x)\ge u\}$, prove the comparison in several small steps (left of $\sup A_u$, right of $\sup A_u$, boundary point, conclusion), and then derive explicitly that $F^{-1}(u)\le x$ if and only if $u\le F(x)$. In proof sections and flashcards, avoid unresolved references such as chapter numbers or lecture numbers unless the target is directly linkable in the repository; prefer self-contained statements or links to the relevant topic-specific note instead. Present the proof in two layers when helpful: first a concrete existence/uniqueness argument using a uniform random variable and half-lines, then the more abstract pushforward view. In the concrete layer, add a short intuition paragraph and matching flashcards explaining what the construction is trying to do, what each side of the key event identity means, what the next proof step will be, and how the concrete proof leads into the abstract proof. In the pseudoinverse layer, when the equivalence proof is multi-step, prefer a nested flashcard cluster whose setup card hints at the next steps and whose conclusion card also explains the intuitive meaning of the equality obtained. In the abstract layer, do not assume the reader already knows the notation: explicitly define the reference measure $\lambda_{(0,1)}$, the preimage notation $T^{-1}(A)$, the pushforward operator $T_{\ast}$ itself, and the resulting notation $T_{\ast}\lambda$ before using them, and whenever the symbol $T_{\ast}$ reappears, restate or expand it enough that the reader does not need to remember the notation from a previous sentence. Also explain _why the pushforward formula is legitimate_: measurability of $T$ makes each preimage $T^{-1}(A)$ measurable, and preimages preserve empty sets, whole spaces, and countable disjoint unions, so $A\mapsto \lambda(T^{-1}(A))$ is genuinely a probability measure rather than just suggestive notation. Likewise, unpack the key half-line identity step by step in both prose and flashcards, preferably as $T^{-1}(( -\infty,x])=\{u:T(u)\le x\}=\{u:F^{-1}(u)\le x\}=\{u:u\le F(x)\}=(0,F(x)]\cap(0,1)$, so readers can see exactly how the abstract generator computation imports the concrete pseudoinverse inequality. Add flashcards there that likewise hint at the next computation and interpret the result obtained. For random-variable notes, explicitly point out where Dynkin's pi-lambda theorem reappears: in the half-line criterion for measurability and in upgrading equality of cumulative distribution functions to equality of laws. For auxiliary set systems such as pi-systems and Dynkin systems, explicitly compare each notion with sigma-algebras, emphasize that pi-systems require only finite intersections, explain why Dynkin systems are built around disjoint unions, and make the disjointification trick readable step by step in both prose and flashcards (e.g. a nested flashcard cluster spelling out the setup, disjointness, union-equality, and the final reason why the pi-system + Dynkin-system structure makes it work). When presenting Dynkin's pi-lambda theorem, add general “how to use it” intuition in both prose and flashcards: identify a small generating pi-system, package the desired property into a Dynkin system, verify the generator is contained in it, and then conclude for the whole generated sigma-algebra. Likewise, for uniqueness-from-agreement arguments, explicitly explain the proof idea that the agreement class is a Dynkin system because complements and disjoint unions preserve equality of measures, and include at least one concrete example such as half-lines or half-open intervals generating $\mathcal{B}(\mathbb{R})$. More generally, do not blindly preserve notation from the source lecture material when a clearer choice of symbols would improve readability; if the existing note already uses a stable notation convention, keep it, but for a fresh proof prefer role-based notation that makes the logic harder to confuse.
   - When introducing circuit notation or jargon (e.g. $V_{CC}$, $V_{CE}$, “low-side switch”), add a brief definition near first use (and optionally a flashcard) so the note is readable without prior course context.
   - When embedding diagrams/schematics, put the image markup (e.g. `<p> ![...](attachments/... )`) on the same line as the preceding paragraph. Add diagram‑recall flashcards (image in the prompt) for key symbols/circuits so learners can recall what each represents. When adding or editing circuit diagram generators (e.g. schemdraw scripts in `attachments/`): draw the main rail first, use `.push()` at the branch node, finish the rail, then `.pop()` and draw the branch; use `.reverse()` on diodes when polarity requires it (see academic-notes skill “Images and circuit diagrams”).
   - IC power pins (e.g. 74HC14): VCC = any valid supply voltage (course may use 5 V); GND = ground. Do not state “must be 5 V” unless the course fixes that.
6. **Session prose**: After the session outline (`---`), add prose only when it adds value (next lecture, grading); do not duplicate index bullets and topic notes. Omit “lecture summary” index cards unless they cover major grading components.
7. **Questions pages** (e.g. `questions.md`, `questions/index.md`, or child pages inside `questions/`): Not topic notes; sections do not need the "Flashcards for this section are as follows:" rubric. Add questions and solutions as content; flashcards are optional per section. For repository-authored review prompts, prefer normal headings/lists rather than markdown blockquotes unless you are quoting actual course material, and explicitly say near the top when the review content is not part of the official course materials.
   - If a questions page mixes official course materials with self-generated review prompts, keep the self-generated prompts in normal list format, but render the official course-material questions as markdown blockquotes. When those quoted course questions need flashcards, place the clozes inside the blockquote so the flashcard viewer shows the full quoted Q&A context.
   - If the file becomes too large, split it into a `questions/` folder with `questions/index.md` plus logically smaller child pages (for example tutorial weeks, midterm review, final review, or solution collections).
   - When official materials naturally come in separate streams (for example tutorial sheets versus problem sets), preserve that structure in `questions/` with separate child pages such as `week N tutorial.md`, `problem set N.md`, or `practice midterm.md` rather than merging unrelated official materials into one page.
   - In official course-material blockquotes, do not add manual numbering in the question line (`> question text`, not `> 1. question text`). If a question has subparts, put those subparts into a markdown list _inside the same blockquote_ (for example `> - (a) ...`, `> - (b) ...`) instead of compressing `(a)`, `(b)`, `(c)` into one long prose sentence.
   - Use `Solution:` instead of `Solution sketch:` and include complete but concise solutions (show the key argument/calculation steps, not just hints). When the question itself has subparts, mirror that structure in the quoted solution with a matching sublist inside the blockquote.
   - Keep cloze coverage high in official course-material blocks: aim to cover almost all key formulas, statements, and final results while preserving readability.
   - When cleaning up low-value clozes in official course-material blocks, do not merely delete them and leave the passage under-covered. If a prompt-side cloze or a symbol-only cloze is not useful, move that coverage into the solution by clozing the method name, interpretation, decision, condition, or final combined result instead.
   - Do not concentrate clozes only on equations; include plaintext clozes for method names, conditions, logical connectors, and interpretations.
   - When auditing coverage in an official quoted solution, check each subpart bullet for three layers: the setup being counted/conditioned on, the method or theorem being invoked, and the final formula/value/conclusion. Thin coverage usually means one of those layers is still fully visible; add clozes there rather than piling more markup onto an already-hidden equation.
   - Strong explanatory pivots are valid flashcard targets. Phrases such as `condition on the random length`, `fix one anchor person`, `the same likelihood ratio reappears`, or `the condition singles out a specific coordinate` are often more valuable than another isolated symbol-only cloze, because they encode why the computation works.
   - In official quoted solutions, avoid bare equation-only clozes of the form `{@{$...$}@}` unless there is a clear reason to test just that formula. In most cases, wrap the explanatory phrase and the equation together in one cloze so the card tests a full reasoning step rather than a floating symbolic fragment.
   - In particular, avoid clozes that only hide a variable name or that merely re-hide a formula already visible in the question prompt. Prefer clozes such as `memoryless property`, `condition on at least one queen`, `switch to $D_2$`, `open intervals`, or an entire final probability statement.
   - After any cloze-cleanup pass on official question pages, explicitly re-audit each quoted solution block (and each subpart bullet, if present) to ensure the block still contains meaningful coverage. A good post-cleanup result usually has at least one textual cloze for the method/interpretation and one cloze for the key final result, rather than only a bare surviving formula cloze.
   - If a tutorial prompt asks for extra detail, include the actual derivation steps in the prose and in the flashcards, not just the final result.
   - For official problem-set and tutorial pages, do not feel bound to reproduce the source solution verbatim. Preserve the correct result and assumptions, but prefer a proof order that is simpler, more intuitive, and easier to learn from when that can be done without losing rigor.
   - When a proof-heavy official solution is being expanded, a good default structure is `main idea -> reduce to a simpler object or split into cases -> carry out the formal step -> state the conclusion clearly`. This often reads better than mirroring the source PDF line-for-line.
   - If an explanatory sentence and its equation belong to one proof step, prefer clozing them together as one reasoning unit rather than making one flashcard for the prose and a separate one for the equation. This is especially useful for countable-sum, conditioning, and reflection-principle arguments.
   - Keep each quoted `Solution:` as one paragraph line (avoid manual soft-wrap line breaks) and avoid suppressing `no_soft_wrap_paragraph` for this style.
   - For consecutive quote blocks, do not suppress MD028 globally; instead, insert `<!-- markdownlint MD028 -->` between adjacent quote blocks, with one blank line before and after the comment. Keep `latex_single_line` unsuppressed by default; if suppression is truly required in dense punctuation contexts, suppress only spacing rules and keep multiline-LaTeX detection active.
   - Never place a cloze marker inside an already-open `$...$` expression (for example, avoid `$P(X)={@{$0.4$}@}$`). If you want to hide an equation, cloze the whole equation as one unit from outside, e.g. `{@{$P(X)=0.4$}@}`.
   - Cloze syntax is strict: use `{@{...}@}` only; do not use `}}` as a close token.
   - Cloze content must stay on one source line, and clozes must not be nested.
   - The validator enforces these as atomic rules: `cloze_open_close_matching`, `cloze_wrong_closing_token`, `cloze_single_line`, and `cloze_no_nested`.
8. Run `uv run .agents/skills/academic-notes/check.py --content <path>` and resolve obvious authoring omissions before opening a PR.
